## The hard counterargument

A network of "what people are doing with their agents" fails if it is just AI-generated changelogs. Most changelogs are boring because the reader is not invested in the project, the stakes are invisible, and the post has no reason to exist after the moment passes.

The product cannot be "everyone else uses their agents this way." That is a curiosity hook, not a durable network.

The product has to answer a sharper question:

> Who is actually building a real solo business with agents, what did the agents do, what changed in the business, and what can I steal for my own work?

## The real object

The object is not a post. The object is a **proof-of-work ledger for agent-run businesses**.

The private object:

- what agents ran
- what they attempted
- what shipped
- what failed
- what human decisions changed the direction
- what external signal arrived: users, revenue, traffic, replies, GitHub issues, demos, sales calls

The public object:

- one weekly story with stakes
- one number or external proof
- one decision
- one reusable tactic
- one artifact link

That is the difference between "my agent edited 12 files" and "I used three agents to test a pricing page, got 14 demo replies, killed the wrong segment, and shipped the narrower offer."

## Why this can work now

The macro tailwind is real: Stripe, Census, and MBO all point in the same direction. More people are forming solo businesses, more independents are reaching meaningful income thresholds, and AI is reducing the need for early hires.

But solo builders have a legibility problem. A team leaves residue: employee profiles, press, GitHub orgs, public roles, job posts, product-marketing pages. A solo operator running agents can create revenue without creating much public evidence. That is the slot Plumcake can own.

Plumcake should not be "social media for agent users." It should be the place where agent-assisted work becomes legible.

## The wedge

Start with builders whose agents are attached to money or distribution:

- solo SaaS founders
- AI-consulting operators
- info-product / newsletter / community operators
- tiny agencies using agents for client work
- productized-service founders
- devtool founders shipping with Codex, Claude Code, Gemini CLI, Cursor, or custom harnesses

Do not start with generic AI enthusiasts. They produce opinions. Operators produce receipts.

## Product shape

### 1. Private first

The user should get value even if nobody else joins.

Plumcake should become the private memory of their agents:

- "What did my agents do this week?"
- "Which threads actually produced a result?"
- "What did I keep re-deciding?"
- "Which project got momentum?"
- "What should I publish?"

If the private ledger is useful, the public network is a natural export. If the private ledger is useless, the feed will be fake.

### 2. Publish only the proof card

The public unit should be a card, not a raw timeline:

- Builder
- Project
- Week
- Agent stack used
- What shipped
- Result / metric / signal
- Decision made
- One tactical lesson
- Evidence links

The card should be short enough to skim and concrete enough to be stolen from.

### 3. Weekly ritual, not infinite feed

Product Hunt worked because shipping became a daily event. Indie Hackers worked because founder stories had stakes. Plumcake's ritual should be weekly:

> Friday agent receipts: what shipped, what moved, what failed.

An infinite feed invites low-stakes noise. A weekly digest forces selection.

### 4. Curation before scale

Open networks decay into spam and performative posting. The first version should be hand-curated:

- 25 operators
- 8 weeks
- one post per person per week
- no public comments at first, or only replies from members
- public read, private write

Manufacture quality before manufacturing volume.

### 5. Paid or qualified access

Payment is not mainly revenue. It is proof that the room is not for tourists.

Good starting options:

- free private Plumcake app
- public read-only gallery
- paid/approved publishing circle

The mistake is charging before the private ledger is useful. The second mistake is letting anyone publish once the gallery exists.

## What to show on the card

Good:

- "Closed $1.2K consulting work after agent-generated audit pass"
- "Killed onboarding step after 9/14 testers got stuck"
- "Claude failed, Codex fallback shipped the patch"
- "Gemini wrote better copy; Codex did the repo surgery"
- "Agent found 37 stale docs links, fixed 31, left 6 manual"
- "Launched a Chrome extension in 4 days; first 19 users from Reddit"

Bad:

- "Worked on auth"
- "Refactored UI"
- "Fixed bugs"
- "Used AI to build faster"
- "My agent made progress"

The card needs stakes, not activity.

## Trust primitives

This category dies if people think the posts are fake AI slop. Plumcake needs proof without forcing users to leak private data.

Useful proof types:

- Git commit / PR link
- deployment link
- screenshot
- revenue range, not exact revenue
- redacted email reply
- issue/ticket link
- agent transcript excerpt
- before/after metric
- "human verified" checkbox

The post should visibly distinguish:

- directly observed by Plumcake
- user-attested
- linked external proof
- redacted/private

That one distinction may matter more than likes.

## The first experiment

Run this as a deliberately small eight-week program.

1. Recruit 25 builders already using agents for real projects.
2. Each connects or manually forwards their agent work logs.
3. Plumcake generates a private weekly summary.
4. User edits one public proof card.
5. Publish a weekly digest of 10-15 cards.
6. Track three metrics:
   - publish rate: how many invited builders publish weekly
   - reader saves/replies: which cards were useful enough to keep
   - imitation: did another builder copy a tactic and report a result?

The strongest signal is not pageviews. It is imitation. If people steal tactics from the cards, the network has value.

## What not to build

- No generic "agents are the future" positioning.
- No open posting at launch.
- No likes as the main status system.
- No raw changelog feed.
- No "here is what everyone is doing" empty curiosity feed.
- No public default for private agent logs.
- No heavy social graph before there is proof content.

## The one-sentence product

Plumcake is a private agent-work ledger that lets serious solo builders publish weekly proof cards showing what their agents actually shipped, learned, earned, or killed.

## Why this is different from Twitter, Indie Hackers, and Product Hunt

Twitter has personality but no structure. Indie Hackers has founder stories but not native agent traces. Product Hunt has launches but not the messy process before launch. Plumcake can own the missing middle: the week-by-week operational record of agent-assisted building.

That is a real slot.

Not "social media for AI agents."

An operating memory that can selectively become public proof.
